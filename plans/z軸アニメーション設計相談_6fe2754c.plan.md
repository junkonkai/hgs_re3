# Z軸アニメーション設計相談

## 現状の整理

- **ツリー表現**: 左上の点から真下に直線が伸び、途中でカーブして少し右の点に接続する動き（X/Y のみ）。帰路（homeward）ではその逆の流れでツリーがたたまれていく。
- **既存の「奥」**: [NodeContentBehind](resources/ts/node/parts/node-content-behind.ts) で、背後ノード（behind-node）に曲線でつなぐ表現があり、opacity や blur で「奥」を表現している。こちらは「同じ階層の別ノードが奥にある」イメージ。

いま検討しているのは、**子ノードの階層そのものを Z 軸の奥に配置し、子リンクをクリックすると「奥にあるノードが現れる／奥へ遷移する」** という動きの追加です。

---

## 参照イメージとレイアウト方針

**実装の参考**: [『エースコンバット7 スカイズ・アンノウン』～ 作中世界の"OS"を表現するGUI ～](https://cgworld.jp/feature/201908-gg2019-ace7.html)（CGWORLD）

- 同作では「メニューの階層が奥行き方向につらなり、縦軸にメニュー項目、横軸にメニューのオプション」という構成で、**階層＝奥行き（Z）** のメタファーが使われている。Z 軸の表現や遷移の雰囲気はこちらに寄せてよい。
- **違いとして維持したい点**: AC7 は横に広がる構成。本プロジェクトでは **スマホ利用を想定** するため、
  - **縦に伸びる**（下方向にツリーが伸びる）構造を維持する。
  - 横スクロールや不自然な改行を避け、縦スクロールで完結させる。
- まとめ: **奥行き（Z）の表現は AC7 に近づける／寄せてよい。広がり方向だけ「横→縦」で現状の下に伸びる構造を維持する。**

## 解釈（確認したい点）

Z の使い方として、少なくとも次の2パターンが考えられます。


| パターン                   | 説明                                                                      |
| ---------------------- | ----------------------------------------------------------------------- |
| **A. 遷移時だけ Z を使う**     | 普段のツリーは今と同じ 2D。子リンクをクリックした「瞬間」だけ、奥から手前に出てくる（または手前から奥へ抜ける）ようなアニメーションにする。 |
| **B. ツリー全体を 3D で配置する** | 子ノードは常に Z 方向に奥に並べておき、クリックでカメラや視点が奥に進む（または奥のノードが手前に出る）ようにする。             |


- **A** は「今の 2D レイアウトはそのまま、遷移のフィードバックだけ Z を足す」イメージ。
- **B** は「階層 = 奥行き」を常に視覚化するイメージ。

どちらに近い（または両方の要素を取りたい）かで、実装の形やコストがかなり変わります。

## 設計上の所感

### 「子＝奥」「遷移＝奥へ進む」について

- **メタファーとして自然**: 「掘り下げる」「深く入る」は階層ナビゲーションと相性が良い。帰路は「手前に戻る」と対応させやすい。
- **既存の「奥」との一貫性**: すでに NodeContentBehind で「奥」の表現があるので、「子の階層も奥」にすると「奥 = 別のノード／別の階層」という一つのルールにまとめられる可能性がある。
- **設計方針（合意）**: 奥にある要素が小さく・かすんで見えるのは許容。「奥になにかがある」と分かれば十分とする。
- **注意したい点**:
  - **アクセシビリティ**: Z 方向の動きは前庭系に負担をかけることがある。`prefers-reduced-motion` では Z アニメーションを無効化し、フェードや単純な切替にフォールバックする想定にしておくと安全。
  - **遷移の主役**: 「何が動いているか」をはっきりさせる必要がある。「視点が奥へ進む」のか「奥のノードが手前に出る」のかで、ユーザーの注意の向きが変わる。

### パターンBの実装方式と検索エンジン（SEO）

パターンBを実装する場合、大きく二通りある。

| 方式 | 内容 | 検索エンジン |
|------|------|--------------|
| **全部 Canvas で描画** | ノードのテキスト・リンクも含め、ツリー全体を canvas 上に描く。 | **不利**。canvas 内のテキストはクローラが通常のドキュメントとして扱わず、インデックスされにくい。 |
| **DOM ＋ transform で制御** | ノードは現状どおり HTML（`section.node`、`node-head`、`<a>` 等）のままにして、`transform: translateZ()` / `translate3d()` や親の `perspective` で Z 方向の位置・スケールを制御する。接続線だけ [CurveCanvas](resources/ts/node/parts/curve-canvas.ts) のまま（必要なら 3D 投影計算を足す）。 | **有利**。見出し・リンク・本文は DOM に残るため、そのままクロール・インデックス対象になる。 |

**結論**: 検索エンジンに引っかかることを気にするなら、パターンBでも **「要素は DOM のまま、translate（transform）で Z を制御」** にするのがよい。全部 canvas に寄せる必要はない。現状もノード本体は HTML・曲線だけ canvas なので、その役割分担を維持しつつ Z を足す形になる。

### 実装コストの目安

- **A（遷移時だけ）**: 子ノード表示時／帰路時に `transform: translateZ()` や `scale` + `opacity` を組み合わせたアニメーションを足す程度。既存の 2D ツリー構造を大きく変えずに試しやすい。
- **B（常時 3D）**: レイアウト計算・ hit test ・接続線の描画を 3D 対応する必要があり、[CurveCanvas](resources/ts/node/parts/curve-canvas.ts) や [ConnectionLine](resources/ts/node/parts/connection-line.ts) まわりとの整合が課題になりやすい。**ノードは DOM＋transform にすれば SEO を維持できる。**

## まとめと次の一歩

- 「子ノードの階層が奥にあり、クリックでそちらに遷移する」というコンセプトは、メタファーとして成立しており、既存の「奥」表現とも合わせやすい。
- 進めるなら、まずは **A（遷移の瞬間だけ Z を使う）** で「奥から手前に出る／手前から奥へ抜ける」ような短いアニメーションを試し、体感とパフォーマンスを見てから、B のような常時 3D を検討するのが現実的だと思います。
- 設計を詰めるうえで、「Z は遷移時だけか、ツリー全体を 3D にするか」と「動くのは視点か、ノード（コンテンツ）か」の2点を決めると、具体的なアニメーション仕様に落とし込みやすくなります。

不明な点や「こういう動きにしたい」という具体像があれば、それを共有してもらえれば、それに合わせてもう一歩だけ踏み込んだ設計（例: キーフレームのイメージ、`internal-node` との役割分担など）も整理できます。
